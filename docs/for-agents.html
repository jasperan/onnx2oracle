For Coding Agents

This page is meant to be pointed at by a human talking to a coding agent. Paste the URL into your agent's context and it'll have everything it needs to install ONNX embeddings into an Oracle AI Database using onnx2oracle.

How to use this page. If you're a human, copy the prompt below and point your agent at this URL. If you're the agent, read top-to-bottom and run the commands in order. Don't skip the preflight.

Quick invocation

Paste this into Claude, Codex, Gemini, or any other coding agent. Replace the bracketed part with your target (local Docker, a DSN you already have, or a Cloud ADB wallet path).

You are installing ONNX embeddings into an Oracle AI Database
using onnx2oracle (https://github.com/jasperan/onnx2oracle).

Read the instruction manual at:
https://jasperan.github.io/onnx2oracle/for-agents.html

Then: <describe your target DB here, e.g. local Docker, Cloud
ADB with wallet at /path/to/wallet, or a specific DSN. Name the
preset model or say "default".>

Follow the manual step by step. Use the 5-preset list unless I
specify otherwise. Run the verify command after loading and
report the results back to me.

Agent preflight

Confirm each of these before running any install commands. If a check fails, stop and tell the user which one, with the exact output.

  1. Python 3.10+ is on the PATH. Run python --version or python3 --version.
  2. pip works. Run pip --version.
  3. Either (a) Docker is running and the user is in the docker group, or (b) the user has a reachable Oracle 23ai or 26ai instance and will provide a DSN.
  4. Internet access is available for HuggingFace downloads. The default model is about 90 MB.
  5. Roughly 2 GB of free RAM for the augmentation step. Peak is short (seconds), not sustained.

Install path A: local Docker

Use this when the user doesn't have Oracle running yet. The first docker up takes 3 to 5 minutes because the PDB is opening for the first time. Don't abort on what looks like a hang.

pip install onnx2oracle
onnx2oracle docker up --wait      # 3-5 min cold start while PDB opens
onnx2oracle load all-MiniLM-L6-v2 --target local
onnx2oracle verify --target local

What success looks like

  • onnx2oracle docker up --wait ends with ready. If it spins past 5 minutes, tail the logs with onnx2oracle docker logs -f.
  • onnx2oracle load ends with model registered as ALL_MINILM_L6_V2.
  • onnx2oracle verify prints VECTOR_EMBEDDING returned 384-d and cosine(self, self) = 1.000000.

Install path B: existing Oracle instance

Use this when the user already has Oracle 23ai or 26ai running (self-hosted, on-prem, or Cloud ADB). Set ORACLE_DSN in the environment; don't hardcode it.

pip install onnx2oracle
export ORACLE_DSN='user/password@host:port/service'
onnx2oracle load all-MiniLM-L6-v2
onnx2oracle verify

Privileges the DB user needs

  • CREATE MINING MODEL
  • EXECUTE ON DBMS_VECTOR
  • SELECT ANY DICTIONARY (or SELECT on user_mining_models)

If the user can't grant these, escalate. Oracle refuses the load without them, and the errors look cryptic (see troubleshooting below).

Preset picker

Five vetted presets. Pick by the user's constraint (speed, quality, long docs, language). If the user says "default", pick all-MiniLM-L6-v2.

Name Dims Size Pooling When to pick
all-MiniLM-L6-v2 384 ~90 MB mean Default. Fastest, smallest. Good for most English search.
all-MiniLM-L12-v2 384 ~130 MB mean Slightly better quality than L6, still fast.
all-mpnet-base-v2 768 ~420 MB mean Best quality among mean-pooled English models.
bge-small-en-v1.5 384 ~130 MB cls Strong retrieval benchmarks. CLS pooling (loader handles it).
nomic-embed-text-v1 768 ~540 MB mean 8192-token context. Best for long documents.

Custom HuggingFace model

If the user wants a model outside the preset list, use --from-huggingface.

onnx2oracle load my-model --from-huggingface sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2
SentencePiece is not supported. Models built on XLM-R, e5, or other SentencePiece tokenizers raise a clear NotImplementedError at augmentation time. WordPiece-based models (BERT, MiniLM, MPNet, BGE) work. If the user needs multilingual, point them at microsoft/Multilingual-MiniLM-L12-H384 as a WordPiece alternative.

Querying once loaded

The canonical one-liner. Substitute the Oracle model name the loader printed.

SELECT VECTOR_EMBEDDING(ALL_MINILM_L6_V2 USING 'hello world' AS DATA) AS v
FROM dual;

Wiring it into a table with a vector column:

CREATE TABLE docs (
  id        NUMBER PRIMARY KEY,
  content   CLOB,
  embedding VECTOR(384, FLOAT32)
);

INSERT INTO docs (id, content, embedding)
VALUES (1, :text, VECTOR_EMBEDDING(ALL_MINILM_L6_V2 USING :text AS DATA));

Dimension must match the preset. 384 for MiniLM and BGE-small, 768 for MPNet and Nomic.

Verifying success

Run onnx2oracle verify (or verify --target local for Docker). It runs four checks. Report the output back verbatim:

  1. Connected. The DSN resolves and the session opens.
  2. Registered. The model is present in user_mining_models.
  3. Shape. VECTOR_EMBEDDING returns the expected dimension count and the L2 norm is approximately 1.0.
  4. Semantics. cosine(king, queen) > cosine(king, banana). If this flips, the pooling layer is wrong for the model.

Good output looks like:

$ onnx2oracle verify --target local
  model: ALL_MINILM_L6_V2
  VECTOR_EMBEDDING(...) returned 384-d ... ok
  cosine(self, self) = 1.000000 ... ok
  cosine(cat, dog)   = 0.672 ... sensible

Troubleshooting

Map the error the user sees to one of these. If the actual message isn't here, check the full troubleshooting page.

ORA-00942: user_mining_models does not exist

Either the DB user lacks SELECT ANY DICTIONARY, or the instance is older than 23ai. DBMS_VECTOR and user_mining_models landed in 23ai. Confirm version with SELECT banner FROM v$version; and escalate if it's 19c or older.

ORA-20000: LOAD_ONNX_MODEL failed

Usually an opset mismatch. Rebuild with onnx>=1.16 and opset 18. The other common cause is an input-tensor name mismatch; Oracle expects the model's text input to be named pre_text. The loader sets this, but a hand-built custom graph might not.

ORA-01031: insufficient privileges

Grant the missing privileges as SYSDBA:

GRANT EXECUTE ON DBMS_VECTOR TO <user>;
GRANT CREATE MINING MODEL TO <user>;

Then retry onnx2oracle load.

NotImplementedError about SentencePiece

Full message looks like: ... SentencePiece ... cannot be represented as a BertTokenizer ONNX graph. The tokenizer-wrapping step only supports WordPiece. Swap to a WordPiece model such as microsoft/Multilingual-MiniLM-L12-H384 and retry.

Fresh install + onnx2oracle docker up hangs

Normal on first run. The PDB cold start takes 3 to 5 minutes. Tail the logs to confirm progress:

onnx2oracle docker logs -f

If logs are silent for more than 10 minutes, the container probably crashed. Check docker ps -a and restart.

Docker permission denied

The user isn't in the docker group. Either add them (sudo usermod -aG docker $USER then log out and back in) or fall back to path B with an existing Oracle instance via ORACLE_DSN.

Cleanup

Drop a loaded model from Oracle:

BEGIN DBMS_VECTOR.DROP_ONNX_MODEL(model_name => 'ALL_MINILM_L6_V2', force => TRUE); END;
/

Stop the local Docker container:

onnx2oracle docker down

Add --volumes to docker down to wipe the PDB data volume too.

Agent rules of thumb

  • Never commit credentials. ORACLE_DSN belongs in the user's environment, not in code.
  • Always run verify right after load. Report the four-line output back verbatim.
  • If the user says "use the default" without naming a model, pick all-MiniLM-L6-v2.
  • If a model with the same Oracle name already exists, ask the user before passing --force.
  • Oracle identifiers are uppercase with underscores. The CLI normalizes HF-style names (e.g. all-MiniLM-L6-v2 becomes ALL_MINILM_L6_V2) automatically.
