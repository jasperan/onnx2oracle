The idea

Oracle 26ai runs ONNX models in-database.

DBMS_VECTOR executes an ONNX graph the same way it executes a PL/SQL function. A string goes in at the SQL layer. A vector comes out.

No external service. No queue. No HTTP hop. The embedding is just another column transformation. 

-- the entire API surface
SELECT VECTOR_EMBEDDING(
  ALL_MINILM_L6_V2
  USING :text AS DATA
) AS v
FROM dual;

-- returns VECTOR(384, FLOAT32)
-- runs inside the DB process
-- no network, no sidecar

Four reasons

Why put the model in the database at all?

01

Zero network latency

No HTTP, no TLS handshake, no queue. The embedding is a local function call. P99 moves from the network to the model itself.

02

No API keys to rotate

Nothing to leak, nothing to expire. Credentials stay inside the same wallet that already governs SQL access.

03

Data never leaves the DB

Document text gets embedded without a byte crossing a network boundary. Compliance reviews get shorter.

04

ACID-native writes

Insert a row, embed it, index it, all inside the same transaction. No dual-writes, no eventual drift, no orphan vectors.

The hard part

HuggingFace ships a Python object.
Oracle wants a single ONNX graph.

A sentence-transformers model on HuggingFace is a Python tokenizer plus a transformer plus a Python pooling function. Oracle needs one self-contained ONNX file that turns a raw string into a pooled, normalized vector, all inside the DB.

Bridging that gap is the whole job.

The augmented pipeline

Every stage the loader bolts on.

Stage 1 — download

Pull the raw ONNX from HuggingFace.

The sentence-transformers repos already ship an ONNX export of the core transformer. No conversion needed at this step. The loader caches under ~/.onnx2oracle/ so repeat loads are instant.

  • MiniLM-L6 is ~90 MB; mpnet-base is ~420 MB; nomic is ~540 MB
  • Hash-verified; partial downloads resume
  • Offline mode reads straight from the cache
from huggingface_hub import hf_hub_download

path = hf_hub_download(
    repo_id="sentence-transformers/all-MiniLM-L6-v2",
    filename="onnx/model.onnx",
    cache_dir="~/.onnx2oracle/hf_cache",
)

# ~90 MB on disk, ready for stage 2
core_model = onnx.load(path)
from onnxruntime_extensions import gen_processing_models
from transformers import AutoTokenizer

tok = AutoTokenizer.from_pretrained(
    "sentence-transformers/all-MiniLM-L6-v2",
)

pre_model, _ = gen_processing_models(
    tok,
    pre_kwargs={},
    opset=14,
)

# pre_model is now a real ONNX graph
# that accepts a STRING input tensor
Stage 2 — tokenizer to ONNX

Wrap the fast tokenizer as ONNX ops.

onnxruntime-extensions emits a prefix graph where BertTokenizer or SentencepieceTokenizer is a real ONNX op, not a Python call. Oracle can execute this.

  • Opset 14 keeps the tokenizer ops compatible with older runtimes
  • BertTokenizer for WordPiece; SentencePiece is noted separately
  • Emits input_ids, attention_mask, token_type_ids
Stage 3 — compose

Bump, copy, merge. In that order.

Before merging, bump the core transformer to opset 18 with onnx.version_converter. Copy the custom opset domains from the pre-model. Then onnx.compose.merge_models stitches the two halves with a pre_ prefix on the inputs.

  • Opset drift is the #1 reason merge fails
  • The pre_ prefix keeps names unique
  • io_map wires tokenizer outputs to transformer inputs
import onnx
from onnx import compose, version_converter

# 1. bump core model to opset 18
core = version_converter.convert_version(core, 18)

# 2. copy custom opset domains from pre-model
for op in pre.opset_import:
    if op.domain == "com.microsoft.extensions":
        core.opset_import.append(op)

# 3. merge with prefix to avoid name collisions
merged = compose.merge_models(
    pre, core,
    io_map=[("input_ids", "input_ids"),
            ("attention_mask", "attention_mask")],
    prefix1="pre_",
)
Stage 4 — pool

Two pooling strategies, one graph each.

The loader reads the model card and picks mean or CLS. Both end as a tiny ONNX subgraph bolted onto the transformer output.

mean pool

Average across the sequence axis. Works for most sentence-transformers models (MiniLM, mpnet, nomic).

cls pool

Take token zero via Gather, then squeeze. Required for BGE and many BERT-family encoders.

Stage 5 — L2 normalize

Divide by the norm.
Guard against zero.

Five small ONNX nodes. Square element-wise, sum along the feature axis, square-root, floor the denominator at 1e-12, divide. The output is a unit vector, ready for cosine similarity in Oracle.

  • Pow → ReduceSum → Sqrt → Max → Div
  • The Max node is the numerical guard — no NaN vectors
̂v = v / max(‖v‖, 1e−12)
where ‖v‖ = √(Σ v2)
Stage 6 — upload

Raw bytes into DBMS_VECTOR.

No filesystem staging, no DIRECTORY object, no OS user. The Python driver holds the ONNX bytes in memory and passes them straight to Oracle as a BLOB bind parameter.

  • Zero disk footprint on the DB host
  • Metadata (dimension, pooling) travels as inline JSON
  • Model name becomes a callable identifier in SQL
cursor.execute("""
    BEGIN
      DBMS_VECTOR.LOAD_ONNX_MODEL(
        model_name => :n,
        model_data => :bytes,
        metadata   => JSON(:meta)
      );
    END;
""", {
    "n": "ALL_MINILM_L6_V2",
    "bytes": augmented_onnx_bytes,  # BLOB
    "meta": json.dumps({
        "function": "embedding",
        "embeddingOutput": "embedding",
        "input": {"string": ["DATA"]},
    }),
})

Known limitation

SentencePiece tokenizers don't round-trip.

Models like intfloat/multilingual-e5-small use SentencePiece. That tokenizer doesn't map cleanly to Oracle's BertTokenizer ONNX op. Rather than emit a broken graph, the loader refuses and suggests a WordPiece alternative.

This is architectural, not a bug. The signal beats the silence.

What users see
NotImplementedError: SentencePiece tokenizer for
  'intfloat/multilingual-e5-small' cannot be
  exported to Oracle's BertTokenizer ONNX op.

  Try a WordPiece model instead:
    bge-small-en-v1.5
    all-MiniLM-L6-v2
    all-mpnet-base-v2
The whole API surface

Once loaded, it's one SQL function.

SELECT VECTOR_EMBEDDING(
  ALL_MINILM_L6_V2
  USING :text AS DATA
) FROM dual;

That's it. No driver changes. No new clients. Every language with an Oracle binding already knows how to call SQL.

DSN resolution

Five layers of precedence, checked in order.

Each layer can shadow the next. The CLI walks them top to bottom and uses the first one that resolves to a connection string.

  1. --dsn <string> Explicit flag. Wins over everything.
  2. $ORACLE_DSN Environment variable, standard Oracle convention.
  3. ~/.onnx2oracle/config.toml Per-user profile with named targets.
  4. --target local Docker-compose defaults (user=racing, pw=racing, port=1521).
  5. Interactive prompt Last resort when nothing else matched.
Security

SQL injection is a boring problem. Treat it that way.

Model names go into VECTOR_EMBEDDING(<NAME> USING …) as a SQL identifier, which must be f-stringed. Everything else uses bind variables. The identifier is validated against a strict whitelist before it's allowed anywhere near a cursor.

  • Pattern: ^[A-Z_][A-Z0-9_]{0,127}$ — no quotes, no semicolons, no spaces
  • Rejection is loud: ValueError with the offending input 
def validate_oracle_name(name: str) -> str:
    if not re.match(
        r"^[A-Z_][A-Z0-9_]{0,127}$",
        name,
    ):
        raise ValueError(
            f"invalid oracle name: {name!r}"
        )
    return name

# >>> validate_oracle_name("FOO) FROM DUAL;--")
# ValueError: invalid oracle name:
#   'FOO) FROM DUAL;--'
